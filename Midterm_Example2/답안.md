카테고리	대상 객체 (반드시 Release 필요)
인프라	Device, DeviceContext, SwapChain

정확하게 짚었어! 네 코드가 틀린 이유와 왜 Release()가 아닌 delete를 써야 하는지, 그리고 교수님이 원하시는 "정답 코드"가 무엇인지 명쾌하게 설명해 줄게.
### 1. 네 코드가 틀린 이유 (오답 분석)
 1. **메모리 해제의 주체 (Release vs delete):**
   * **Release()**는 보통 **DirectX(COM 객체)**에서 사용하는 방식이야. 내부적으로 참조 횟수(Reference Count)를 줄이다가 0이 되면 스스로를 파괴하는 방식이지.
   * 하지만 지금 네가 작성 중인 **Component 클래스**는 일반적인 C++ 클래스고, new를 통해 생성되었어. C++에서 new로 만든 건 반드시 **delete**로 지워야 메모리가 실제 시스템으로 반환돼. Release()라는 함수는 현재 Component 클래스에 정의되어 있지도 않을 가능성이 커.
 2. **배열 관리의 문제:**
   * 메모리만 지운다고 끝이 아니야. GameObject가 사라질 때 그 안의 포인터 배열(vector)에 담긴 주소들이 가리키는 실제 메모리를 하나하나 찾아가서 delete 해줘야 해.
### 2. 왜 delete인가? (COM이 아니라서?)
맞아! **"COM 객체가 아니기 때문"**이야.
 * **DirectX 객체(ID3D11Device 등):** 시스템 레벨에서 관리되므로 Release()를 호출.
 * **일반 C++ 객체(GameObject, Component 등):** 개발자가 직접 new 했으므로 delete로 책임지고 삭제.
### 3. 교수님이 원하시는 "정답 코드"
교수님이 "배열 size만큼 돌면서 delete 하라"고 하셨다면, 표준적인 vector 순회 삭제 코드를 작성해야 해.
```cpp
~GameObject() 
{
    // 1. 벡터의 크기만큼 반복문을 돌며
    for (int i = 0; i < (int)components.size(); i++) 
    {
        // 2. 실제 메모리 해제 (컴포넌트 파괴)
        // 이때 Component의 소멸자가 virtual(가상 소멸자)이어야 자식들까지 깨끗하게 지워짐!
        if (components[i] != nullptr)
        {
            delete components[i]; 
        }
    }
    // 3. 포인터 주소들이 담긴 벡터를 비움 (선택사항이지만 깔끔함)
    components.clear();
}

```
### 📝 시험 적중 포인트: 가상 소멸자 (Virtual Destructor)
이 문제와 세트로 무조건 나오는 개념이 있어. 바로 **Component 클래스의 소멸자 앞에 virtual이 붙어있는가?** 하는 점이야.
 * **왜?** components 배열에는 PlayerControl 같은 자식 클래스들이 담겨 있잖아.
 * 만약 부모인 Component의 소멸자에 virtual이 안 붙어 있으면, delete components[i]를 했을 때 부모 부분만 지워지고 **자식(PlayerControl)이 가진 데이터는 메모리에 남아서 누수가 발생해.**
> **시험 대비 답변:** "부모 클래스인 Component의 소멸자를 가상 소멸자(virtual)로 선언해야, 자식 클래스들의 메모리까지 안전하게 해제할 수 있습니다."
> 
### 💡 요약하자면
 * new로 만든 일반 객체는 **delete**로 지운다.
 * vector를 돌면서 각 요소(Component*)를 하나씩 delete 한다.
 * 부모 클래스(Component)의 소멸자는 반드시 **virtual**이어야 한다.
1000줄짜리 코드에서 소멸자 부분 빈칸이 나오면 이 로직을 그대로 적으면 돼! 👊 이제 헷갈리지 않지?

네가 올려준 코드를 다시 확인해봤어. **결론부터 말하면 Component 클래스에는 virtual이 아주 잘 붙어있어!**
하지만 이 부분이 왜 중요한지, 그리고 교수님이 시험에서 어떤 식으로 뒤통수를 칠 수 있는지 정확히 짚어줄게.
### 1. 코드 속 virtual 확인
올려준 코드의 25~31행 부근을 보면 이렇게 되어 있어.
```cpp
class Component
{
public:
    class GameObject* pOwner = nullptr;
    bool isStarted = false;

    // [확인] 가상 함수들 (자식에서 재정의 가능)
    virtual void Start();
    virtual void Update(float dt);
    virtual void Input();
    virtual void Render();
    
    // [핵심] 가상 소멸자! (자식의 메모리까지 해제하기 위한 장치)
    virtual ~Component(); 
};

```
**가상 소멸자(virtual ~Component())**가 선언되어 있기 때문에, GameObject 소멸자에서 delete components[i];를 호출하면 부모인 Component만 지워지는 게 아니라, 실제 객체인 PlayerControl의 메모리까지 깨끗하게 지워지는 거야.
### 2. 시험에 나올 수 있는 "버그 찾기" 시나리오
만약 1000줄짜리 코드에서 교수님이 **"메모리 누수가 발생한다. 원인을 찾으시오."**라고 문제를 낸다면, 가장 유력한 범인은 **virtual 키워드의 누락**이야.
 * **상황:** Component의 소멸자에 virtual이 빠져 있다.
 * **결과:** GameObject에서 delete를 해도 자식 클래스인 PlayerControl의 소멸자는 호출되지 않는다.
 * **정답:** "부모 클래스(Component)의 소멸자가 가상 소멸자가 아니어서 다형성을 통한 자식 객체의 완전한 해제가 불가능합니다. 소멸자 앞에 virtual을 추가해야 합니다."
### 3. 왜 Release() 대신 delete인가? (다시 강조!)
네 코드에서 Release()가 오답이었던 이유는 **두 가지 약속**이 다르기 때문이야.
 1. **C++ 표준 약속 (new / delete):**
   * 일반적인 클래스 설계 방식.
   * 메모리를 직접 관리함.
 2. **DirectX/COM 약속 (AddRef / Release):**
   * 스마트 포인터처럼 내부에서 숫자를 세서 관리함.
   * ID3D11Device 같은 객체들에만 해당됨.
**"네가 직접 만든 클래스(Component, GameObject)는 C++ 표준 객체니까 무조건 delete로 지워야 한다!"**는 게 교수님의 의도야.
### 💡 요약: 소멸자 빈칸 채우기 정답지
만약 시험에 빈칸으로 나온다면 이렇게 적어!
```cpp
~GameObject() 
{
    // 1. 배열 요소를 하나씩 확인하며 삭제
    for (int i = 0; i < (int)components.size(); i++) 
    {
        if (components[i] != nullptr) 
        {
            delete components[i]; // virtual 소멸자 덕분에 자식까지 삭제됨
            components[i] = nullptr; // 안전을 위해 포인터 초기화
        }
    }
    // 2. 벡터 자체를 비움
    components.clear();
}

```
